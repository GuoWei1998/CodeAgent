# AI 项目学习方法：简单可执行版

适合学习 Claude Code、AI Agent、命令行工具这类小项目。

核心原则：

> 不要逐行抄代码，也不要直接让 AI 全写完。  
> 每次只学一个小闭环，然后自己改一点。

## 每次学习就按这 5 步走

### 1. 先跑通

目标：先知道项目能干什么。

做法：

- 跟着教程把项目跑起来。
- 不要求完全理解。
- 不要在细节里卡太久。

完成标准：

- 项目能正常运行。
- 你知道输入是什么、输出是什么。

### 2. 写 5 句话总结

关掉教程，用自己的话写：

```md
这个项目的目标是：
用户输入：
程序中间做了：
程序输出：
最核心的 1 个函数/模块是：
```

写不出来就回去看一眼，再关掉继续写。

这一步比抄代码重要。

### 3. 只重写最小版本

不要完整复刻教程，先写最小可运行版本。

例如 Claude Code 类项目，最小版本可以是：

```text
用户输入一句话
-> 调用一次大模型
-> 打印回复
```

先别加：

- 工具调用
- 多轮循环
- 文件读写
- 错误处理
- 漂亮的代码结构

完成标准：

- 你能从空文件写出一个能跑的小版本。

### 4. 每次只加一个功能

按照这个顺序加：

```text
第 1 步：用户输入 -> 模型回复
第 2 步：加 read_file 工具
第 3 步：加 write_file 工具
第 4 步：加 run_command 工具
第 5 步：加多轮循环
第 6 步：加错误处理
```

规则：

- 一次只加一个功能。
- 加完就运行测试。
- 出错就记下来。

不要一口气做完整项目。

### 5. 做一个小改造

学完后必须改一点，不然容易只是“看懂了”。

可以选一个最小改造：

- 把 `read_file` 改成 `list_files`
- 把命令行输入改成读取配置文件
- 把单轮回复改成多轮对话
- 把固定 prompt 改成可配置 prompt
- 把项目改成 README 生成器
- 把项目改成代码解释助手

完成标准：

- 你的项目和教程不完全一样。

## AI 应该怎么用

不要这样问：

```text
帮我写一个完整项目
```

更推荐这样问：

```text
我在学习一个简化版 Claude Code 项目。
请不要直接给完整代码。
请把它拆成 5 个小任务，每个任务只给目标和验收标准。
```

或者：

```text
我先自己写了一版代码。
请你只做 code review，指出 bug、结构问题和我没理解好的地方。
不要直接重写。
```

或者：

```text
请问我 5 个问题，检查我是否真的理解了这个项目的架构。
先不要给答案。
```

AI 的最佳角色：

- 教练
- 出题人
- 代码评审
- 卡住时的提示器

AI 不适合一直当：

- 代写员

## 最小学习记录模板

每次学项目时，新建一个 `learning.md`，只写这些：

```md
# 学习记录

## 项目目标


## 最小版本


## 核心流程


## 今天卡住的地方


## 我做的小改造


## 下次要补的点

```

不要写太长。  
每个部分 1 到 3 句话就够。

## 一个推荐节奏

### 第一天

- 跑通教程。
- 写 5 句话总结。
- 不追求完全理解。

### 第二天

- 不看原代码，重写最小版本。
- 只实现“用户输入 -> 模型回复”。

### 第三天

- 加一个工具，比如 `read_file`。
- 让 AI 做 code review。

### 第四天

- 加第二个工具，比如 `write_file` 或 `run_command`。
- 记录一个最容易出错的点。

### 第五天

- 做一个小改造。
- 从头说一遍项目流程。

## 判断有没有学会

如果你能做到这 4 件事，就算这轮学得不错：

- 不看教程能讲清项目流程。
- 能从空文件写出最小版本。
- 能说出哪里最容易出 bug。
- 能做一个和教程不同的小改造。

## 最重要的一句话

每个教程不要追求“完整学完”，而是追求：

> 跑通一次，复述一次，重写一个最小版，改造一个小功能。

这样比逐行抄代码更快，也比完全交给 AI 更能真正学进去。
